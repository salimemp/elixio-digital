# Internationalization (i18n)

> 42 locales, 9 fully translated priority languages, cookie-based SSR with no flash of English.

---

## Overview

Elixio Digital ships with internationalization in **42 locales** across 5 continents. The system uses cookie-based locale resolution (not URL prefixing) so all language variants serve from the same canonical URL — better for SEO since `hreflang` annotations handle language signaling without splitting ranking signals across `/en/about` vs `/es/about`.

| Tier | Locales | Status |
| --- | --- | --- |
| Priority (native translation) | `en`, `es`, `fr`, `de`, `hi`, `pt`, `ar`, `ur`, `he` (9) | ✓ Translated, marked `needs_human_review: false` |
| CJK (machine-translated, high traffic) | `zh`, `zh-TW`, `ja`, `ko` (4) | ✓ Translated, marked `needs_human_review: false` |
| Scaffolded (English fallback) | `bn`, `bg`, `ca`, `cs`, `da`, `el`, `et`, `fi`, `hr`, `hu`, `id`, `it`, `lt`, `lv`, `ms`, `nl`, `no`, `pl`, `ro`, `ru`, `sk`, `sl`, `sv`, `ta`, `te`, `th`, `tr`, `uk`, `vi` (29) | English fallback, `needs_human_review: true` |

Total: **42 locales**.

## Architecture

```
                          ┌─────────────────────────────┐
                          │ apps/web/src/lib/i18n.ts      │
                          │ - LOCALES (42)                │
   Browser cookie  ─────► │ - resolveLocaleFromCookie()  │
   localStorage          │ - loadMessages() (server)    │ ◄── messages/<locale>.json
                          │ - formatPrice / formatNumber │
                          │ - makeT() for t(key)         │
                          └────────────┬────────────────┘
                                       │
                          ┌────────────▼────────────────┐
                          │ apps/web/src/lib/i18n-client │ (client-side)
                          │ - I18nProvider               │
                          │ - useI18n() hook             │
                          │ - CLIENT_MESSAGES (13)       │ ◄── en/es/fr/de/hi/pt/ar/he/ur/zh/zh-TW/ja/ko
                          │ - setLocale() persists cookie│
                          └─────────────────────────────┘
```

### Why no URL prefixing (`/en/about`)?

1. **SEO**: One URL per content piece = one ranking signal. Google uses `hreflang` to serve the right language version of the same URL.
2. **Backwards compat**: We don't have to redirect legacy URLs.
3. **Cookie + localStorage**: Persists across navigations and gives instant client-side switching.

### Why client bundles only the 13 most-used locales?

Static import of all 42 message bundles would add ~40 KB to the JS bundle. Instead we statically import the 9 priority + 4 CJK (= 13) and rely on server-side rendering for everything else. Scaffolds fall back to English client-side via `loadClientMessages()` → `en` if not found.

## Locale resolution

Priority order:

1. **`locale` cookie** (set by `LanguageSwitcher` when user picks a language).
2. **`elixio-locale` localStorage** (set on first visit for fallback).
3. **`DEFAULT_LOCALE`** = `"en"`.

Server-side (in root layout):

```ts
const cookieStore = cookies();
const locale: Locale = resolveLocaleFromCookie(cookieStore.get("locale")?.value) ?? DEFAULT_LOCALE;
```

The server renders the page in the resolved locale. The client hydrates with `initialLocale` from the server, so there's no flash of English on first paint.

## Adding a new locale

1. Create `apps/web/messages/<locale>.json` with `_meta`:

   ```json
   {
     "_meta": {
       "name": "Spanish",
       "code": "es",
       "native_name": "Español",
       "direction": "ltr",
       "currency": "EUR",
       "needs_human_review": false,
       "translator": "Elixio core team",
       "reviewed_by": "Native speaker name",
       "reviewed_at": "2026-06-28"
     },
     "footer": { ... },
     "cookie": { ... }
   }
   ```

2. Add the locale to `LOCALES` in `apps/web/src/lib/i18n.ts`.
3. Add to `LOCALE_CURRENCY` if it has its own currency (e.g. `ja: "JPY"`).
4. Add to `RTL_LOCALES` if it's RTL (currently `ar`, `he`, `ur`).
5. (Optional) Add to `LOCALE_INTL` if it needs a non-standard BCP 47 tag (e.g. `"zh-TW": "zh-Hant"`).
6. For high-traffic locales (>= 1M speakers), add to `CLIENT_MESSAGES` in `i18n-client.tsx` for instant client switching.
7. Add footer + cookie translations.

## Currency formatting

`formatPrice(cents, currency, locale)` uses `Intl.NumberFormat` with the locale's BCP 47 tag:

| Locale | Currency | Example (10000 cents) |
| --- | --- | --- |
| `en` | USD | $100.00 |
| `es` / `fr` / `de` | EUR | 100,00 € |
| `ja` | JPY | ￥10,000 (no decimals — JPY is smallest unit) |
| `ko` | KRW | ₩10,000 |
| `zh` | CNY | ¥100.00 |
| `zh-TW` | TWD | NT$100.00 |
| `hi` | INR | ₹100.00 |

For locales that have multiple valid currencies (e.g. users in `en` who pay in EUR), the `formatPrice(cents, currency, locale)` accepts an explicit currency. The default per-locale currency comes from `currencyForLocale(locale)`.

## Number formatting

`formatNumber(n, locale)` uses `Intl.NumberFormat`. Most locales use standard comma grouping (1,234,567). Chinese, Japanese, Korean use the same comma grouping but render in their respective digit shapes — visually the same.

## RTL support

Right-to-left locales (`ar`, `he`, `ur`) flip the document direction via `document.documentElement.dir = "rtl"` in the I18nProvider. CSS uses `dir="rtl"` selectors for layout flips (e.g., `padding-inline-start` instead of `padding-left`). We do **not** use CSS Logical Properties exclusively — Tailwind's `rtl:` variant covers the few directional utilities we use (margins, paddings, borders).

## Date / time formatting

`formatRelativeTime(date, locale)` uses `Intl.RelativeTimeFormat({ numeric: "auto" })` — e.g. "5 minutes ago", "il y a 5 minutes", "5分前".

## Translation status per locale

See the `_meta` block in each `apps/web/messages/<locale>.json`. When `needs_human_review: true`, a native speaker should review the strings before marketing in that locale.

### Translation priority order (when adding human translators)

1. The 9 priority locales with `needs_human_review: false` — already reviewed.
2. The 4 CJK locales — high traffic, machine-translated, need native review.
3. Spanish variants (es-MX, es-AR, etc.) — currently grouped under `es`.
4. Portuguese variants (pt-BR) — currently grouped under `pt`.

## Legal pages and translation notice

The legal pages (`/privacy`, `/terms`, `/cookies`) are server components that render in English by default. A `<TranslationNotice>` component (`apps/web/src/components/legal/TranslationNotice.tsx`) shows a yellow banner at the top of each legal page when the user's locale is non-English, telling them English is the canonical legal text and the translation is for convenience only.

**Why we don't auto-translate legal pages**: Legal text requires professional translation (often via a sworn translator for enforceability). Machine translation introduces risk of meaning drift. The notice is the right user experience — it gives confidence that the binding text is in a known language, while still being welcoming in the user's preferred language.
